# Karpathy 4-Phase Checklist

코딩 작업을 시작·진행·종료할 때 다음 4단계를 순서대로 적용한다.

---

## 1. FRAME — 코드 쓰기 전

먼저 다음을 출력하고, 불확실한 것이 있으면 답을 기다린 뒤 진행한다.

- **이해한 요구**: 한 줄 요약
- **가정 (≤3개)**: 명시적으로 적기
- **불확실한 것**: 해석이 2개 이상 가능하면 코드 작성 전 사용자에게 질문
- **DONE-WHEN**: 검증 가능한 완료 조건 리스트

---

## 2. WRITE — 최소 코드

사용자가 명시적으로 요청하지 않은 다음 패턴은 추가하지 않는다.

- try/except, 에러 핸들링, 입력 검증
- 1회용 코드의 클래스화·과도한 함수 분리
- "나중을 위한" 옵션 파라미터·기본값
- 추상 인터페이스·래퍼·팩토리

작성 후 자가 점검 — **삭제 테스트**:

> "이 줄을 지워도 동작하는가? → 지워라."

---

## 3. EDIT — 외과적 수정

- 같은 파일 안에서도 무관한 줄은 건드리지 않는다.
- 자동 포매터가 무관한 줄을 변경했다면 수동으로 되돌린다.
- 변경 직전 예상 diff 라인 수를 선언: "이 변경은 ~N줄 예상".
- 본인이 만든 죽은 코드만 본인이 치운다. 남의 dead code는 별도 PR로 분리.

---

## 4. VERIFY — 루프

작업 시작 시 다음 3줄을 선언한다.

    GOAL:       <한 줄>
    DONE WHEN:  <검증 가능한 조건 리스트>
    VERIFY BY:  <명령어 또는 관찰 방법>

루프:

    실행 → 검증 → (실패 시) 가설 수정 → 재실행

**3회 연속 실패 = 멈추고 사용자에게 보고**한다. 같은 가설로 4회째 시도하지 않는다.

---

## 충돌 처리

프로젝트 `CLAUDE.md`가 명시적으로 다른 규칙을 두면 **프로젝트 규칙을 우선**한다.

예: 프로젝트가 "에러 처리는 항상 추가한다"는 정책을 명시하면, 2단계의 try/except 금지보다 프로젝트 규칙을 따른다.
